创建对象

虽然使用Object构造函数或对象字面量都可以用来创建单个对象，但这些方式有个明显的缺点：使用同一个接口创建很多对象，会产生大量的重复代码。为解决这个问题，人们开始使用以下一些创建对象的模式。

工厂模式

工厂模式解决了创建多个相似对象的问题，但却没有解决对象识别的问题（即怎样知道一个对象的类型）。

构造函数模式

与工厂模式不同之处：

  1. 没有显式地创建对象；

  2. 直接将属性和方法赋给this对象；

  3. 没有 return

按照惯例，构造函数命名始终以一个大写字母开头， 创建一个新对象，必须使用 new 操作符，构造函数模式可以将他的实例标识为一种特定的类型，而这正是胜过 工厂模式 的地方。

构造函数模式不足之处：

构造函数的每个方法都要在每个实例上重新创建一遍，因此不用实例上的同名函数是不相等的。

原型模式

person1 、person2 作为 Person 对象的实例，person1中存在一个内部属性，指向构造函数的原型对象，因此 person1 、person2 共享了原型对象上的属性和方法。

属性屏蔽

在实例中添加一个属性，而该属性与实例原型中的一个属性同名，那我们就在实例中创建该属性，该属性将会屏蔽原型中的那个属性。

使用 delete 操作符可以删除实例上的属性，解除屏蔽。

原型模式的不足之处：

原型中的所有属性是被实例共享的，基本数据类型可以 属性屏蔽 ，但是 引用数据类型 不存在 属性屏蔽。

组合使用构造函数和原型模式

这种继承了原型模式的特性，又解决了所有实例共享原型对象 引用数据类型属性 的问题。

动态原型模式

动态原型模式避免了原型方法的重复定义。

寄生构造函数模式

依旧没有解决对象识别的问题。

原型链

在 javaScript 中，每个对象都有一个指向它的原型（prototype）对象的内部链接。这个原型对象又有自己的原型，直到某个对象的原型为 null 为止（也就是不再有原型指向），组成这条链的最后一环。这种一级一级的链结构就称为原型链（prototype chain）。

基于原型链实现继承

这种方式的问题在于：

  1. 通过父类实例重写子类原型对象实现继承，父类的实例属性被转为子类的原型属性，因此父类中引用类型的实例属性会被所有子类的实例共享。
  2. 在创建子类实例时，不能向父类的构造函数中传递参数。

通过下面 借用构造函数 的方式可以解决。

组合继承

组合继承，有时候也叫做伪经典继承，指的是将原型链和借用构造函数的技术组合到一块，从而发挥二者之长的一种继承模式。主要思路是利用 原型链实现对原型属性和方法的继承 ，而通过 借用构造函数来实现对实例属性的继承。

组合继承模式避免了原型链和借用构造函数模式的缺陷，融合了他们的优点，成为了JavaScript中最常用的继承模式

原型式继承

在只想让一个对象与另一个对象保持类似的情况下，原型式继承是完全可以胜任的。不过，问题用原型链实现继承一样，引用类型的属性始终都会共享相应的值。

寄生式继承

寄生继承就是不用实例化父类了，直接实例化一个临时副本实现了相同的原型链继承。常用于 非构造函数式继承 。

寄生组合式继承

组合继承是JavaScript最常用的继承模式，不过，它也有不足，组合继承最大的问题就是无论什么情况下，都会调用两次父类型构造函数。

寄生组合式继承既有组合继承的优点，又避免了两次调用父类构造函数的性能浪费。
